Layer System Plan — Review & Critique
Date: February 25, 2026
Reviewer: Grok (xAI)
Document Reviewed: LAYER_SYSTEM_PLAN.md (Phase 16 proposal)
Overall Assessment
The Layer System plan is a natural and high-value evolution of the Layout Builder delivered in Phase 15. It directly addresses real usability friction identified after implementation (tab switching, mental z-index tracking, inconsistent background handling) and aligns perfectly with the Photoshop/Figma mental model that creative marketing users expect. The decision to deliver a non-breaking UI projection in Phase 16a is technically excellent — it delivers 90–95 % of the perceived benefit with near-zero migration risk and full backward compatibility.
The plan is concise, well-scoped, realistically estimated (5–6 days for 16a), and includes thoughtful open questions. With minor refinements noted below, this feature will make the Layout Builder feel like a professional design tool rather than an advanced form. It is ready for implementation with the adjustments recommended.
Critiques

Cross-Type Reordering (Open Question 1)
The plan correctly identifies this as the core value proposition yet leaves it as an open question. Restricting reordering to within-type in 16a would significantly dilute the benefit and create a frustrating “almost there” experience. Users will expect to drag a slot above an overlay (or vice versa) and see the visual stacking update immediately. The extra implementation cost is modest once normalizeZIndices() is made cross-array.

Background as a First-Class Layer (Open Question 2)
The plan proposes moving background controls to the Properties Panel when the background row is selected. This is correct architecturally but creates a discoverability risk: background properties are currently always visible in a dedicated tab. Making the background row selectable is the right long-term choice, but the transition must be communicated clearly in the UI (e.g., a tooltip on first open or a one-time onboarding banner). [developer comment: We don't need to worry about communicating this, users have not used the app yet, it's still pre-release]

Slot Visibility Scope (Open Question 3)
The visible flag is introduced without specifying its runtime behaviour. If a hidden slot is omitted from the finalized gallery, it affects lightbox indexing and thumbnail-strip behaviour in viewport mode — a non-trivial side effect. For Phase 16a the flag should be builder-only (a “draft / hide in editor” feature). Exposing it to the public gallery should be deferred to Phase 16b or 17 when the full elements array exists.

Drag Implementation in LayerPanel
The plan mentions “React DnD or simple mousedown-based swap”. React DnD is heavy for a simple vertical list reorder. The existing codebase already uses react-rnd; introducing another dragging library would contradict the lean philosophy. HTML5 native Drag and Drop API (with draggable=true + dataTransfer) is sufficient, battle-tested, and zero-dependency. [Developer comment: we do already have react-rnd in use, why couldn't we leverage that?]

Naming and Persistence
The plan correctly notes that names must persist once users start renaming layers. However, the data-model addition (name?: string) is listed under both LayoutSlot and LayoutOverlay without clarifying the default fallback logic (“Slot N”, “Overlay N”, or “Background”). This must be explicit to avoid inconsistent labelling across the canvas, layer panel, and properties panel.

Locked State in Finalized Rendering
The plan applies locked only inside the builder (disableDragging on Rnd). This is correct for 16a, but the documentation should explicitly state that locked has no effect on the public gallery. Future phases may want a “freeze layout” toggle that respects locked slots for client preview purposes.

Minor Scope Creep Risk in 16a
Moving background controls to the Properties Panel and adding a collapsible Media Picker sidebar are both improvements, but together with the full left-panel replacement they constitute a non-trivial UI reshuffle. Visual regression risk is correctly flagged as “Medium” in the effort table; the plan should include a specific regression checklist (slot selection highlight, panel widths, responsive modal behaviour). [Developer comment: we definitely should do this switchover, but carefully as suggested]

Suggestions
High-Priority (Implement Before Starting 16a)

Resolve Open Questions Explicitly
Cross-type drag reorder: Allowed in 16a. Implement normalizeZIndices() to operate on a temporary merged list, then write the new z-index values back to the respective slots and overlays arrays.
Background selectable: Yes. Show background properties in the right panel when selected; keep a small persistent “Background” chip at the top of the Properties Panel for quick access during transition.
Slot visibility: Builder-only for 16a. Document clearly: “Hidden slots are visible in the finalized gallery; use this flag for temporary hiding while designing.”
Name fallback: Define in types/index.ts:TypeScriptgetLayerName(item: LayerItem): string {
  if (item.name) return item.name;
  if (item.kind === 'slot') return `Slot ${item.index}`;
  if (item.kind === 'overlay') return `Overlay ${/* calculated index */}`;
  return 'Background';
}

Adopt Native Drag-and-Drop for Layer Reordering
Use the HTML5 Drag and Drop API inside LayerRow.tsx. It requires ~40 lines of code, works with keyboard (accessibility win), and keeps the bundle unchanged. Reserve a heavier library only if multi-select layer dragging is added later.
Enhance buildLayerList() for Stability
Add a stable sortKey (e.g., original array index as tie-breaker) so that reordering does not cause unexpected jumps when z-indices are equal after a drag. Include this in the ~15 unit tests already planned.

Medium-Priority (Nice-to-Have in 16a)

Undo/Redo Integration
Since Phase 15 already uses Immer (or structuredClone) for history, expose the new layer actions through the existing undo stack. Users expect Ctrl+Z after reordering or renaming a layer.
Ghost Mode for Hidden Layers
Extend the existing 10 % opacity ghost mode (already present for overlays) to hidden slots. This gives immediate visual feedback in the canvas without affecting finalized rendering.
Keyboard Shortcuts for Layer Panel
Align with Phase 15 accessibility additions:
↑ / ↓ to navigate rows
Space to toggle visibility
L to toggle lock
F / B for Bring to Front / Send to Back


Low-Priority / Future

Phase 16b Consolidation
Defer until usage data shows frequent cross-type z-index conflicts (e.g., via the telemetry framework from Phase 14). The projection approach in 16a is sufficiently powerful for the first 12–18 months.

Conclusion & Recommendation
The Layer System plan is strong, low-risk, and will meaningfully elevate the Layout Builder’s perceived quality. With the resolutions to the open questions and the lean drag implementation noted above, it can be delivered cleanly in a single 5–6 day sprint.
Sign-Off: Approved with modifications. Proceed to implementation once the four high-priority open questions are documented in the plan and the native drag-and-drop choice is confirmed. The feature will ship as a highlight of Phase 16 and will be the logical next step after the solid foundation built in Phase 15.
